# Project case study writing

A case study's job is to make the reader's "I'd interview this person" reflex fire. Show thinking, not screenshots.

Structure each case study:
1. **One-line hook** — what it is and why it's interesting.
2. **Problem** — the real constraint or challenge (concrete, not "I wanted to learn X").
3. **Approach** — key decisions and the tradeoffs behind them. This is the part recruiters' engineers actually read.
4. **Outcome** — result, quantified where possible (latency, users, time saved, bundle size).
5. **Stack** — tech used, briefly.

Rules:
- Specific over impressive. Numbers over adjectives.
- Make at least one real tradeoff visible.
- Write into the typed content file in `content/`, not into a component.
- Keep it skimmable: bold the hook, short paragraphs.

For voice/polish, hand off to the `content-editor` agent.
